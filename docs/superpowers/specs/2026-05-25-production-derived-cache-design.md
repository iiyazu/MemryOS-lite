# Production Derived Cache Design

## Purpose

Upgrade the current Redis recall-cache probe into a production-grade derived
cache layer for MemoryOS Lite recall/context reads.

This design keeps MemoryOS Lite as an eval-driven, source-attributed Agent/RAG
memory prototype. It does not make Redis authoritative, does not change the
default memory architecture, and does not promote benchmark scores as goals.

## Scope

In scope:

- Query analysis cache.
- Recall candidate cache.
- Recall context package cache.
- Cache diagnostics surfaced through metadata.
- Redis and no-op cache backends.
- Conservative invalidation through memory watermarks plus TTL.

Out of scope:

- Embedding cache.
- Qdrant or ANN query cache.
- v1 `ContextBuilder` caching.
- Agent kernel write-path caching.
- Benchmark score optimization.
- Production auth, deployment, or multi-service operations.

## Existing Baseline

Current behavior is already SQLite-first:

- SQLite is the authoritative store for sessions, messages, episodes, pages,
  items, core memory, archival memory, traces, and source refs.
- Page files and traces are debug mirrors.
- Default memory architecture is v3.
- v1 remains available with `MEMORYOS_MEMORY_ARCH=v1`.
- v2 recall remains opt-in with `MEMORYOS_RECALL_PIPELINE=v2`.
- The agent kernel remains opt-in with `MEMORYOS_AGENT_KERNEL=v1`.

The current Redis probe stores reconstructable JSON values and falls back to
SQLite recomputation on Redis errors. It needs a stronger schema, key contract,
validation, diagnostics, and invalidation policy before it can be treated as a
production-grade derived cache layer.

## Design Principles

- Redis is never memory authority.
- Cache entries are disposable derived results.
- Cache miss, stale, corrupt, timeout, or Redis outage must not change answer
  correctness.
- Source refs and diagnostics must survive cache round trips.
- Cache status must be observable in metadata.
- Default behavior must remain unchanged without Redis and when cache is
  disabled.
- Cache keys must be generated by a single builder, not hand-built by business
  code.
- Invalidation must be conservative: watermark changes force a new key, and TTL
  is a fallback safety net.

## Architecture

Add a `DerivedCache` layer between the SQLite-backed store and the recall/context
read path:

```text
SQLite authoritative memory
  -> MemoryWatermarkProvider
  -> RecallPipeline / ContextComposer computation
  -> DerivedCache optional read/write
  -> ContextPackage + diagnostics + source_refs
```

Components:

- `DerivedCache`: backend-neutral interface for `get`, `set`, `delete`, and
  `status`. It returns typed results and does not leak Redis runtime exceptions.
- `CacheEntry`: common envelope for all derived payloads.
- `CacheKeyBuilder`: the only supported way to produce cache keys.
- `MemoryWatermarkProvider`: produces a revision snapshot from SQLite state.
- `RedisDerivedCache`: Redis adapter. It stores and loads envelopes but does not
  understand recall semantics.
- `NoopDerivedCache`: default backend. It preserves behavior when Redis is not
  configured or cache is disabled.
- `CacheDiagnostics`: structured status included in `ContextPackage.metadata`
  and v3 layer metadata.

First integration points:

- `query_analysis`
- `recall_candidates`
- `recall_context_package`

## Cache Entry Schema

All backend values use an envelope:

```json
{
  "schema_version": 1,
  "entry_version": "derived-cache-v1",
  "scope": "recall_context_package",
  "created_at": "2026-05-25T00:00:00Z",
  "ttl_s": 300,
  "authority": {
    "store": "sqlite",
    "store_revision": "watermark-sha256",
    "source": "derived"
  },
  "fingerprint": {
    "memory_arch": "v3",
    "recall_pipeline": "v2",
    "settings_hash": "sha256...",
    "watermark_hash": "sha256...",
    "query_hash": "sha256...",
    "parameters_hash": "sha256..."
  },
  "payload": {
    "type": "context_package",
    "data": {}
  },
  "diagnostics": {
    "source_refs_count": 3,
    "planned_ids_count": 3
  }
}
```

Payload rules:

- `query_analysis` stores stable `QueryAnalysis` fields.
- `recall_candidates` stores candidate ids, rank metadata, and diagnostics. It is
  not final context authority by itself.
- `recall_context_package` stores `ContextPackage.model_dump(mode="json")` and
  must be restored with `ContextPackage.model_validate()`.

## Cache Key Schema

Key shape:

```text
{namespace}:{entry_version}:{scope}:{fingerprint_hash}
```

Changing `{namespace}` is treated as a full cache miss. Existing keys do not
need migration because Redis entries are disposable derived results.

The fingerprint hash is derived from structured inputs:

- scope
- memory architecture
- recall pipeline
- settings hash
- memory watermark hash
- query hash
- session id when scope is session-bound
- budget and other scope parameters

Scope parameters include budget, limit, neighbor policy, candidate count, and
any future scope-specific retrieval parameters that can change the derived
result.

Business logic passes structured inputs to `CacheKeyBuilder`; it does not
construct Redis keys directly.

## Invalidation

Invalidation uses two layers.

Strong invalidation:

- `MemoryWatermarkProvider` reads SQLite state and produces a revision snapshot.
- The watermark hash is included in the key fingerprint.
- Memory mutations select a new key automatically.
- Old keys can remain in Redis until TTL expiry.

The initial watermark covers:

- messages
- episodes
- pages
- items, including content digest
- core memory
- archival documents
- archival chunks
- archival passages
- archival memories

TTL fallback:

- `query_analysis`: 1 hour default.
- `recall_candidates`: 5 minutes default.
- `recall_context_package`: 5 minutes default.
- TTL prevents indefinite stale reuse if watermark coverage is incomplete or a
  future provider has a bug.

## Read Validation

On cache read, the caller must validate:

- envelope is JSON object
- schema version matches
- entry version matches
- scope matches current request
- fingerprint matches current request
- payload type matches expected scope
- `ContextPackage` payload validates when used
- source refs and planned ids have basic structural consistency

Any validation failure becomes `stale`, `corrupt`, or `invalid` diagnostics and
falls back to SQLite recomputation.

## Write Policy

Write only after successful deterministic computation.

Allowed writes:

- successful query analysis
- successful recall candidate planning
- valid context package

Do not write:

- invalid payloads
- partially validated cache hits
- exceptions

Budget-truncated results may be written only if the truncation state is explicit
in payload and diagnostics. Empty results may be written only if diagnostics
explain that the empty result was computed, not caused by cache failure.

## Error Handling

Redis failures never escape into recall or composer control flow.

Read outcomes:

- `hit`: payload validated and returned
- `miss`: recompute from SQLite
- `stale`: recompute and overwrite
- `corrupt`: recompute and overwrite
- `error`: recompute and record fallback reason
- `disabled`: bypass cache

Write outcomes:

- `stored`
- `disabled`
- `error`

Write failures are recorded in diagnostics and do not affect the returned
context package.

## Observability

Cache diagnostics are added to metadata only. Required schema fields do not
change.

Example:

```json
{
  "cache": {
    "enabled": true,
    "backend": "redis",
    "scope": "recall_context_package",
    "status": "hit",
    "key_version": "derived-cache-v1",
    "watermark_hash": "sha256...",
    "latency_ms": 2.1,
    "fallback_reason": null,
    "write_status": "stored"
  }
}
```

Compatibility:

- Preserve existing `recall_cache` metadata.
- Preserve existing `query_analysis_cache` metadata.
- v3 layer metadata may include cache diagnostics.
- Do not add a metrics system in this phase.

## Settings

Settings remain opt-in:

- `MEMORYOS_REDIS_URL`
- `MEMORYOS_RECALL_CACHE_ENABLED`
- `MEMORYOS_CACHE_NAMESPACE`
- `MEMORYOS_CACHE_DEFAULT_TTL_S`

Production-grade settings should add scoped TTLs:

- `MEMORYOS_CACHE_QUERY_ANALYSIS_TTL_S`
- `MEMORYOS_CACHE_RECALL_CANDIDATES_TTL_S`
- `MEMORYOS_CACHE_CONTEXT_PACKAGE_TTL_S`

Defaults must preserve current behavior:

- Redis URL unset means no-op backend.
- Recall cache disabled means no reads or writes.
- v1 fallback is unaffected.
- Kernel remains opt-in.

## Testing

Unit tests:

- `NoopDerivedCache` returns disabled and does not read/write.
- `CacheKeyBuilder` changes key when settings, watermark, query, session, or
  parameters change.
- envelope schema mismatch returns stale.
- corrupt JSON returns corrupt.
- scope mismatch returns stale.
- Redis adapter round-trips JSON and TTL.
- Redis outage returns error without exception.
- cached `ContextPackage` restores source refs and diagnostics.
- message mutation changes watermark/key.
- episode mutation changes watermark/key.
- page mutation changes watermark/key.
- item content mutation changes watermark/key.
- core memory mutation changes watermark/key.
- archival memory mutation changes watermark/key.
- metadata includes hit, miss, stale, error, fallback reason, and write status.

Integration tests:

- Docker Redis smoke: miss -> write -> hit.
- Docker Redis outage: correct context still returned.
- v2 recall with cache hit preserves source refs.
- v3 composer receives cache diagnostics in layer metadata.
- `MEMORYOS_MEMORY_ARCH=v1` and no Redis are unchanged.

Gate commands:

```bash
uv run pytest tests/test_memory_cache.py tests/test_recall_cache.py -q
uv run pytest tests/test_derived_cache.py -q
uv run ruff check src/memoryos_lite/cache tests/test_derived_cache.py
uv run mypy src/memoryos_lite/cache src/memoryos_lite/retrieval/recall_pipeline.py
```

Full suite remains the final merge gate. Public benchmarks remain diagnostics
and must not become score targets.

## Rollout Plan

1. Introduce `DerivedCache`, `CacheEntry`, `CacheKeyBuilder`, and
   `MemoryWatermarkProvider` while preserving current Redis behavior. The first
   watermark implementation must use lightweight aggregate queries such as
   `count` and `max(updated_at)` or `max(rowid)` where available; it must not
   perform full-table content hash scans except for the existing item-content
   digest required to detect item text mutation.
2. Migrate current v2 recall cache to use the new contract.
3. Add diagnostics compatibility fields so existing metadata consumers keep
   working.
4. Add Docker Redis smoke tests and fake-backend unit tests.
5. Document production cache semantics in `docs/store-interface.md` or a new
   cache guide.

## Non-Goals

- Do not make Redis authoritative.
- Do not cache writes.
- Do not lower source attribution requirements.
- Do not change benchmark scoring semantics.
- Do not make v3 kernel default.
- Do not remove v1 fallback.

## Open Risks

- Watermark generation can become expensive if it scans large tables. The first
  production iteration should keep it correct; a later iteration can introduce a
  dedicated revision table.
- Query analysis cache is global by query/settings and must avoid storing
  user-visible text in keys.
- Context package payloads may grow; TTL and scope-specific payload limits may
  be needed later.
