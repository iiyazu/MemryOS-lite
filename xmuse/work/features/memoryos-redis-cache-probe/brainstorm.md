## Options Considered

1. Full recall-path cache probe in this turn.
   - Benefit: demonstrates miss/hit behavior for v2 recall.
   - Cost: touches behavior-critical retrieval and requires more verification
     around source refs, stale rejection, and unchanged defaults.

2. Phase 0/1 cache contract and optional Redis adapter.
   - Benefit: bounded, testable, default-off, and creates the safe API surface
     needed for later recall wiring.
   - Cost: does not yet measure repeated recall performance.

3. Config-only placeholder.
   - Benefit: smallest change.
   - Cost: does not prove serialization, failure fallback, optional dependency,
     or key safety.

## Selected Slice

Use option 2. Adopt the existing Phase 0/1 work, remove generated bytecode, add
the missing deterministic key builder, and verify with focused tests and ruff.

## Design Notes

- Redis is represented by a `MemoryCache` protocol with `NoopMemoryCache` and
  `RedisMemoryCache`.
- `create_memory_cache()` returns no-op when Redis is unset, unavailable, or
  cannot be constructed.
- Redis payloads are schema-versioned JSON envelopes; corrupt or incompatible
  entries are reported as misses/stale/corrupt diagnostics for recompute.
- `build_cache_key()` is pure and does not connect to Redis. It hashes queries
  so keys do not leak raw message/task text.
- The key builder accepts a caller-supplied memory watermark string; the later
  recall/store phase can define the SQLite-derived watermark source.
