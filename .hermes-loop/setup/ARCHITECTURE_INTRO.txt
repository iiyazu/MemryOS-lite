================================================================
  Hermes Loop — 四节点自动循环开发状态机
================================================================

## 一句话

4 个专用 AI Agent（God / Plan / Execute / Review）通过文件契约接力，
从蓝图到代码全自动迭代，God 动态调整方向，Review 拿指标把关。

================================================================
  四个节点
================================================================

1. GOD (上帝)
   持有: blueprint.md (项目蓝图)
   职责: 读蓝图 → 拆 phase → 写 god_dispatch.json
         → 收 ACK → git commit 存档 → 反思 subagent → 可能调整蓝图 → 推进
   调整: 收 escalate → 分析 subagent → 决定拆分/放宽/重设/放弃 → 更新蓝图
   存档: 每 phase 通过后自动 git commit，可随时回滚
   Superpowers: writing-plans, kanban-orchestrator, brainstorming

2. PLAN AGENT (计划者)
   接收: god_dispatch.json
   产出: brainstorm.md → spec.md → plan.md → plan_final.md
   流程: brainstorming (subagent) → 起草 → 自审查 (subagent) → 迭代定稿
   Superpowers: writing-plans, brainstorming, requesting-code-review

3. EXECUTE AGENT (执行者)
   接收: plan_final.md
   产出: result.md → execute_review.md
   流程: subagent-driven × N task → TDD (RED→GREEN→REFACTOR) → 内审
   修复: 收到 Review FAIL → 只修指出的问题 → 重新提交
   Superpowers: subagent-driven-dev, TDD, systematic-debugging, code-review

4. REVIEW AGENT (审查者)
   接收: god_dispatch.json (指标) + result.md (执行结果)
   产出: ack.json (PASS) / review_verdict.json (FAIL) / escalate.json (升级)
   流程: 对照 God 指标逐项检查 → 安全扫描 → benchmark → 决策
   升级: iter ≥ 3 仍 FAIL → escalate → God 介入
   Superpowers: requesting-code-review, evaluating-llms-harness

================================================================
  文件契约 (所有通信走文件)
================================================================

| 文件                   | 生产者    | 消费者    | 作用                  |
|------------------------|----------|----------|----------------------|
| blueprint.md           | 你 (人)  | God      | 项目总蓝图             |
| state.json             | 全局     | 全局     | 状态机当前状态          |
| god_dispatch.json      | God      | Plan + Review | task + 指标      |
| brainstorm.md          | Plan     | Plan     | 多方案对比             |
| spec.md                | Plan     | Plan     | 设计文档              |
| plan.md                | Plan     | Plan     | 实现步骤              |
| plan_final.md          | Plan     | Execute  | 最终计划              |
| result.md              | Execute  | Review   | 执行报告              |
| execute_review.md      | Execute  | Review   | Execute 内审          |
| review_verdict.json    | Review   | Execute  | FAIL + 修复指导        |
| ack.json               | Review   | God      | PASS 验收确认          |
| escalate.json          | Review   | God      | 3 次失败升级           |
| reflect_{phase}.md     | God      | God      | 反思报告 + 调整建议     |
| escalate_analysis_{phase}.md | God | God      | 升级分析 + 替代方案     |
| git commits            | God      | 人/git   | 每 phase 通过自动存档   |

================================================================
  关键设计决策
================================================================

- 所有通信走文件，无 API，无消息队列 —— 节点之间完全解耦
- 每个节点独立 Codex 窗口运行，读 state.json 决定自己要做什么
- God 可以动态调整蓝图，不绑定初始计划
- Review 对照 God 的指标而非自己的标准 —— 指标由人定义
- Execute 不制定计划，Plan 不写代码，Review 不批准自己审查的代码
- 3 次迭代失败自动升级到 God，不无限循环

================================================================
  你的角色
================================================================

你只需要:
1. 写 blueprint.md —— 你要什么，一个 phase 一段
2. 看 ack.json —— God 推进到哪了，Review 怎么说
3. 必要时介入 —— 改蓝图，调整指标，或直接终止

其余全是自动的。
