## Findings

### High - Movement status has no executable baseline source

References:

- `.hermes-loop/work/phase-2/context_bundle.md` lines 61, 211-222
- `.hermes-loop/work/phase-2/spec.md` lines 57, 147-159
- `.hermes-loop/work/phase-2/plan.md` lines 277-282, 459-486

The phase explicitly requires `pass_to_fail`, `fail_to_pass`, and unchanged-case reporting so regressions are not hidden behind aggregate pass rate. The plan defines `baseline_verdict` as an optional argument and defaults missing baseline data to `new_case_no_baseline`, but it does not specify where baseline verdicts come from, how they are loaded, or which tests prove non-`new_case_no_baseline` movement statuses.

As written, execute_lane could wire all cases with `baseline_verdict=None`, produce required-looking fields, and still hide pass-to-fail regressions. Task 12 asks for movement lists, but the implementation tasks do not create the comparison input needed to populate them.

Required fix:

- Add a concrete baseline-comparison source and wiring step, for example from `.hermes-loop/work/phase-0/baseline_case_matrix.md`, a previous report artifact, or an explicit phase-local comparison map generated by the milestone analysis.
- Add RED tests proving `pass_to_fail`, `fail_to_pass`, `unchanged_pass`, and `unchanged_fail` can be produced from known baseline/current verdict pairs.
- Specify how missing baseline data is reported without satisfying the anti-demo movement requirement.

### High - RED tests do not prove `retrieval_miss` and `evidence_hit_answer_fail` cannot collapse

References:

- `.hermes-loop/work/phase-2/context_bundle.md` lines 58-63, 211-222
- `.hermes-loop/work/phase-2/god_dispatch.json` lines 87-92, 102-110
- `.hermes-loop/work/phase-2/spec.md` lines 60-68, 147-159
- `.hermes-loop/work/phase-2/plan.md` lines 48-118, 152-166

The required RED evidence is specifically that `retrieval_miss` and `evidence_hit_answer_fail` cannot collapse into one generic status. The Task 1 test allows the evidence-hit case to be either `supported_cited_answer` or `evidence_hit_answer_fail`, so it does not force an answer-failure path. The Task 2 source-hit separation test only asserts the failure is not `retrieval_miss`; it can pass with `unsupported_answer` and still never prove the `evidence_hit_answer_fail` class.

Required fix:

- Add a RED test with one case whose expected evidence is retrieved/selected/rendered and whose projected or judged answer fails, asserting `failure_class == "evidence_hit_answer_fail"`.
- In the same test or a paired fixture, assert a missing-evidence case is `retrieval_miss`, and assert the two classes are distinct.
- Keep `unsupported_answer` covered separately so answer-support failures do not replace the required evidence-hit-answer-fail contract.

### Medium - Partial-report schema compatibility is stated but not tested

References:

- `.hermes-loop/work/phase-2/spec.md` lines 119-125
- `.hermes-loop/work/phase-2/plan.md` lines 19-22, 125-150, 292-310

The plan says to preserve partial/final JSON compatibility, but its RED tests only inspect `to_report()` on returned results and do not force the partial-report write path to include `case_diagnostics` and mirror fields.

Required fix:

- Add a focused test or execution check that exercises the partial-report path and asserts the same diagnostic schema as the final report for at least one case.
